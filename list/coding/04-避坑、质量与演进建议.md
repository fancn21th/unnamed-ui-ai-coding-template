# 04 · 避坑、质量与演进建议

本文汇总 **如何减少人类返工**、**本案例踩过的坑**，以及 **中长期架构方向**（配置化、协议化）。与 [《02-对话模拟与状态机制》](./02-对话模拟与状态机制.md) 的「高频修正点」互补：02 偏实现细节，04 偏流程与路线图。

---

## 1. 减少返工：需求与协作侧

| 建议 | 说明 |
|------|------|
| **设计稿前置** | 尽量提供 Figma 文件与关键节点；空状态、加载态、完成态分开说明。 |
| **写清交互时序** | 用短句描述：**先流式 → 何处阻塞 → 用户点什么继续 → 再流式**；比只说「要智能一点」少很多轮对话。 |
| **写清状态规则** | 例如：「父步骤在子步骤未全完成前不得 success」「报告阶段结束后思考条须收口」。 |
| **分阶段开发** | 先 **空状态 + 发送 + 单段流式**；再加 **一处阻塞**；再加 **多步与子项分批**。每步验收后再堆复杂度。 |

---

## 2. 减少返工：工程与规则侧

| 建议 | 说明 |
|------|------|
| **沉淀「对话 mock」约定** | 在 Rules 或 Skill 中写明：推荐目录（config / runner / hook / 展示映射）、阶段类型、`generating` 与内容 prop 的惯例。 |
| **避免 phaseIndex 硬编码蔓延** | 展示层大量写死下标时，改剧本极易漏改；应逐步改为 **phaseId 映射表** 或 **数据驱动子组件**。 |
| **文档与代码同步** | 阶段含义、阻塞点列表、与设计稿的对应关系，建议在迭代中 **随手补在团队 wiki 或本套 list 文档**，降低接手成本。 |

---

## 3. 本案例「做得好的」与「待改进的」（复盘摘要）

### 3.1 做得好的

- **模块化**：模拟独立、类型清晰，配置集中，调整节奏与文案多数不动核心循环。
- **组件复用**：思考步骤、任务列表、表单、报告等尽量用库内 composed，视觉一致。
- **渐进修正**：问题描述具体（「某步过早 success」）时，AI 迭代快。

### 3.2 待改进

- **首次生成偏乐观**：易漏空状态、生成态内容回传、步骤完成边界，需要人类补验收标准。
- **阶段与 UI 耦合**：部分展示逻辑依赖具体索引，扩展成本高。
- **Mock 与真接口**：若协议不一致，对接时可能 **重写驱动层**；应提前想清边界（见下文演进）。

---

## 4. 演进路线（短期 → 长期）

### 4.1 短期（低成本）

- 将延时、文案、阶段顺序 **尽量留在配置文件**，避免散落在多个 `setTimeout` 里。
- 在 PR / 评审单中强制检查：**新增 block 类型是否三处同步（类型 / 配置 / UI）**。

### 4.2 中期

- **配置驱动**：JSON/YAML 描述阶段序列，UI 只做解析渲染；非程序员也可调文案与节奏（需校验工具）。
- **Mock 与 API 同构**：抽象同一套「事件」：`stream_update`、`block_wait`、`block_confirm`、`done`；mock 与 SSE 只是数据源不同。

### 4.3 长期

- **协议先行**：后端与 mock 输出同一对话协议；前端 **渲染器** 只认协议，不认「是不是 mock」。
- **工具化**：可视化编阶段、录制真实会话导出 mock、协议校验测试。

---

## 5. Code Review 检查清单（对话类需求）

- [ ] 初始无多余用户消息；空状态是否符合设计/产品描述  
- [ ] 流式阶段结束时再进入阻塞，阻塞确认后 **不重复** 已播放过的 intro  
- [ ] `generating` 态下用户可见内容与 `streamingText`（或等价状态）一致  
- [ ] 思考步骤：子项节奏、父级完成时机、报告结束收口  
- [ ] 新增阶段是否避免仅靠 `phaseIndex === n` 维护（若已大量存在，新增需求是否顺带重构一小段）  
- [ ] 未在通用组件库内为单页改默认样式（除非已走评审）
