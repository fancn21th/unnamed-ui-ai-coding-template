# 02 · 对话模拟与状态机制

本文面向 **要改剧本、调流式节奏、查「为什么不往下走」** 的开发者：说明阶段类型、Hook 状态含义、与 UI 的衔接方式，并收录本案例里 **高频的人类修正点**（从复盘提炼，不依赖外部文档路径）。

---

## 1. 阶段类型（剧本原语）

一条「演示剧本」是 **有序阶段数组**。本案例使用四类（可扩展，但需同步类型、配置与 UI）：

| 类型 | 作用 | 典型字段 |
|------|------|----------|
| **text** | 打字机式流式输出 | `content`，可选 `charsPerTick`、`tickInterval` |
| **delay** | 停顿，制造「逐步出现」感 | `delayMs` |
| **block** | 暂停，等用户点击确认/提交 | `blockType`：`task_list` / `confirm_panel` / `supplement_form` 等 |
| **instant** | 立即进入下一阶段，常用于触发 UI 侧子更新而不打字 | `render` 或占位（实现上可为 `null`） |

**约定**：`blockType` 字符串与对话面板里 `switch (blockType)` **必须一致**；新增一种阻塞 = 同时改 **类型定义 + 配置 + 渲染分支**。

---

## 2. Hook 状态字段（概念）

下列字段共同驱动「助手消息里显示什么、是否可点确认」。（具体命名以宿主代码为准，此处为语义说明。）

| 状态 | 含义 |
|------|------|
| `phaseIndex` | 当前执行到阶段数组的下标 |
| `phaseId` | 当前阶段 id，便于日志与调试 |
| `streamingText` | 当前 text 阶段已流式输出的片段 |
| `isStreaming` | 是否正在流式 |
| `isBlocked` | 是否在等用户处理阻塞 UI |
| `blockType` | 阻塞类型，供面板选择组件 |
| `isDone` | 整段剧本是否结束 |
| `isGenerating` | 广义「生成中」：含流式 + 阻塞等待 |

**对外方法（典型）**：`sendMessage` 开始跑剧本；`confirmBlock` 结束当前阻塞并进入下一阶段；`reset` 清空重来。

---

## 3. UI 衔接方式

- **消息列表**：用户消息由面板直接插入；助手侧在「生成中」时常用 **单条助手消息 + 动态内容**。
- **流式文案**：组件库若区分 `generating` 状态，需注意 **必须把流式中的内容传到「生成中可见」的 prop**（本案例曾出现：状态为 generating 但内容走默认空，导致「发了消息却看不到 AI」——复盘归类为 **组件用法** 而非业务逻辑错误）。
- **阻塞卡片**：仅在 `isBlocked && blockType === '…'` 时渲染对应卡片；确认后只调 `confirmBlock`，不要在面板里手写 `phaseIndex++`。
- **思考步骤 / 子项分批**：往往由 **多个 delay + instant** 或 UI 内局部状态配合完成；评审点：**父步骤何时算 loading、何时算 success**（见下一节）。

---

## 4. 本案例高频修正点（从复盘提炼）

下列问题在人类验收中 **反复出现**，新剧本开发建议提前在需求/流程说明里写清。

### 4.1 初始与空状态

- 进入页时 **不要** 预置用户消息；空状态应按设计做欢迎区与可点击建议（若有稿）。
- 消息列表组件需支持 **自定义空状态** 插槽，避免写死在库里。

### 4.2 流式与「生成中」展示

- `generating` 时必须把当前片段传到组件要求的 **生成中内容** 入口，否则界面空白。

### 4.3 步骤与子步骤节奏

- 子步骤 **不要一次性全部 success**；需要 **子项间延时**、**逐项 loading → success**。
- **父步骤** 在子步骤未全部完成前 **不应** 显示完成态；报告整合步骤在整段结束前要 **正确收口** 为完成。

### 4.4 体验

- 流式与延时 **默认往往偏快**，需按产品观感调高 `ITEM_DELAY`、`TOOL_DELAY`、`STEP_DELAY` 类常量（或配置化）。

---

## 5. 扩展新剧本的推荐步骤

1. 复制最小示例（见 `examples/minimal-phases.example.ts`）理清 **text → block → text** 闭环。
2. 在配置中 **分段追加**，每加一类 `block` 先在面板加渲染分支并手动点通。
3. 再加重思考步骤、多段 delay；**每加一档复杂度就验收一轮**，避免一次堆几十阶段难以调试。

---

## 6. 已知架构债（评审可点名）

- **阶段索引与 UI 映射**：若展示层大量 `phaseIndex === 42` 式硬编码，新增阶段易漏改；中长期应收敛为 **按 phaseId 或阶段 type 查表**，或 **协议化事件流**（见 [《04-避坑、质量与演进建议》](./04-避坑、质量与演进建议.md)）。

---

## 7. 小结

| 主题 | 要点 |
|------|------|
| 剧本 | 数组顺序即时间线；block 必须人工确认才前进 |
| 状态 | 流式、阻塞、完成三类布尔要语义一致 |
| 返工热点 | 生成中内容回传、父子步骤完成时机、节奏快慢 |
