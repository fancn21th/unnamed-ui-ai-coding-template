# 03 · AI 协作与工程化底座

本文面向 **技术负责人与评审**：说明在本案例中 **AI 编码有效的前提**、**工具链分工**、**投入产出观察**。内容从「简要总结 / 复盘」类材料提炼，写法保持自包含，便于单独上传 `list/` 包阅读。

---

## 1. 为什么说「底座」很重要

AI 生成代码时，若没有统一栈、组件来源和设计依据，容易 **幻觉 API** 或 **风格漂移**。本案例效率较高的前提是：已有 **模板化布局 + 业务组件库 + 明确规则 + 可查 MCP**。

| 底座项 | 作用 | 对 AI 的直接帮助 |
|--------|------|-------------------|
| **预设工作台布局** | 三栏、顶栏、折叠等骨架已就绪 | 对话实现聚焦在「中栏」，少写布局胶水代码 |
| **业务组件库 + 设计 Token** | 消息、思考步骤、卡片、表单等可组合 | 生成代码有明确 import 目标与样式约束 |
| **Cursor Rules** | 技术栈、组件优先级、少改库、页面质量 | 输出与仓库约定一致，减少 Review 扯皮 |
| **Skills（如生成页面）** | 拆需求 → 蓝图 → 选型 → 实现顺序 | 复杂页面时有固定套路，降低对话来回次数 |
| **组件库 MCP** | 查 props、示例、目录 | 减少「猜组件名猜错」导致的运行期错误 |
| **Figma 类 MCP** | 节点、截图、设计上下文 | 空状态、间距、层级等 **有依据**，减少纯口头描述 |

**一句话**：底座回答 AI 三个问题——**用什么组件、按什么规范、长什么样**。

---

## 2. 工具链分工（编码阶段）

| 工具类型 | 典型用途 | 人类需要做的配置 |
|----------|----------|------------------|
| **Rules** | 自动注入栈、别名、composed/blocks/ui 顺序、Token、改库约束 | 维护规则文件，与真实工程一致 |
| **Skills** | 大页面/新模块时按步骤生成 | 写好 SKILL 与示例，培训团队触发语 |
| **组件 MCP** | 查 API、减少瞎写 | 配置可访问的组件源路径或服务 |
| **Figma MCP** | 拉稿、对齐空状态与细节 | 本机 Figma、权限、控制选区避免超限 |

---

## 3. 本案例中 AI 与人类的分工（实证向）

### 3.1 AI 擅长落地的部分

- 三栏内 **页面拼装**、**对话模拟** 的类型与配置、`streamText` / `delay` 等工具函数草稿。
- 在 **MCP 可查** 的前提下，**消息列表、思考步骤、阻塞卡片** 等组合代码。
- 根据指令 **调延时、改文案、修明显 Bug**。

### 3.2 人类修正集中的部分

- **流程与状态**：何时 loading、何时 success、子项是否分批、阻塞后是否重复 intro。
- **组件契约细节**：例如生成态必须把内容传到正确的 prop（见 [《02》](./02-对话模拟与状态机制.md)）。
- **体验节奏**：默认偏快，需要人类拍「够慢、可看」。

### 3.3 观察结论（供领导汇报）

- **开发速度**：相较从零手写，核心 UI + mock **在数轮对话内可到可演示程度**。
- **设计还原**：**有 Figma + MCP** 时，空状态与布局类返工 **相对较少**。
- **复杂度上限**：流式 + 阻塞 + 多步 + 子项联动叠加后，**逻辑类返工轮次仍可能较多**，适合用配置化/协议化逐步治理（见 [《04》](./04-避坑、质量与演进建议.md)）。

---

## 4. 与「只写提示词」的差别

仅有对话而没有 Rules/MCP 时，AI 仍可能写出 **能跑但不符合本库习惯** 的代码。本案例的边际收益来自：

- **规则**固定栈与分层；
- **MCP** 把「查文档」从人类搬运变成机器可查；
- **Skill** 把「大任务拆步」产品化。

推行时建议：**先补规则与组件可查性，再追求提示词技巧**。

---

## 5. 小结（给领导的三个 bullet）

1. **AI Coding 在本案例有效**，前提是布局与组件库已模板化，并配有 Rules +（可选）MCP。  
2. **产出**为可演示的流式对话 mock、阻塞与多步思考，**模拟与 UI 分层**利于后续接真接口。  
3. **主要成本**在状态与时序验收，而非堆界面；长期应走 **配置化 / 协议化** 降复杂度。
