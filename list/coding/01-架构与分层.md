# 01 · 架构与分层

本文说明本案例工程中 **代码如何分层**：哪里放「壳」、哪里放「页面」、哪里放「对话模拟」、哪里尽量 **不要动**。便于评审时判断职责是否清晰、后续替换接口时改动面有多大。

---

## 1. 分层总览

```
┌─────────────────────────────────────────────────────────────┐
│  应用壳（顶栏 + 三栏布局）                                     │
│  只负责挂载左 / 中 / 右三个「页面模块」，不写对话业务细节          │
├─────────────────────────────────────────────────────────────┤
│  页面模块（数据来源 / 对话 / 工作空间）                          │
│  组合业务组件库（composed / blocks），处理本栏交互与展示          │
├─────────────────────────────────────────────────────────────┤
│  对话面板内部：展示层 + 模拟驱动                                │
│  · 展示：消息列表、空状态、流式内容、阻塞卡片、思考步骤等            │
│  · 驱动：独立「对话模拟」模块（类型、阶段配置、runner、Hook）        │
├─────────────────────────────────────────────────────────────┤
│  业务组件库（wuhan 等）+ 基础 UI（shadcn 等）                   │
│  优先通过业务页面 className / 组合扩展，避免为单页改库内默认实现      │
└─────────────────────────────────────────────────────────────┘
```

**原则**：**模拟逻辑与 UI 解耦**。换真实 SSE/WebSocket 时，理想情况是 **主要改「驱动层」**，消息渲染组件尽量复用。

---

## 2. 各层职责（对照本案例）

### 2.1 应用壳

- **做什么**：全局布局、三栏宽度/折叠策略、顶栏品牌与入口占位。
- **不做什么**：不写某条对话文案、不写阶段索引推进。
- **价值**：AI 生成业务页时 **不必再从零搭骨架**，减少风格漂移。

### 2.2 页面模块（左 / 中 / 右）

- **左栏**：数据来源列表、上传区等（演示期多为静态或 Mock）。
- **右栏**：工作空间、报告卡片占位等。
- **中栏（对话）**：最复杂；内部再分为 **容器组件**（发送、状态订阅）与 **内容构建**（根据模拟状态拼出助手气泡里的内容：Markdown、思考步骤、阻塞组件等）。

### 2.3 对话模拟模块（建议独立目录）

本案例将下列职责收拢到同一逻辑模块（名称示例：`chat-simulation`）：

| 文件职责（概念） | 说明 |
|------------------|------|
| **类型** | 阶段联合类型：`text` / `delay` / `block` / `instant` 等 |
| **配置** | 有序阶段数组：文案、流式参数、延时、阻塞类型、占位 instant |
| **纯函数** | 流式逐字输出、Promise 延时，便于单测 |
| **Hook** | 维护 `phaseIndex`、是否流式、是否阻塞、`confirmBlock` 后继续等 |

**边界**：Hook **不** 直接 import 具体卡片 UI；阻塞类型用字符串枚举，由 **对话面板** 根据类型渲染对应组件（待办清单、确认面板、补充表单等）。这样换剧本时优先改 **配置 + 少量分支**，而不是改 Hook 核心循环。

### 2.4 业务组件库与基础 UI

- **composed / blocks**：消息、思考步骤、任务卡片、表单、报告卡片等，应 **大量复用**。
- **修改策略**：与常见团队约定一致——**先业务层 `className` / 组合**；确需通用能力再提需求改库，避免单页污染默认值。

---

## 3. 数据流（简化）

1. 用户点击发送 → 对话面板调用模拟 Hook 的「开始」方法。
2. Hook 按 **阶段数组** 顺序执行：`text` 调 runner 流式更新状态 → `block` 置 `isBlocked` 并等待 `confirmBlock` → `delay` 用定时器推进 → `instant` 用于触发仅 UI 侧关心的瞬时更新。
3. 对话面板（或子组件）**订阅状态**，把 `streamingText`、当前 `blockType`、步骤索引等映射为 **MessageList / 自定义气泡内容**。

评审时可问：**「若下一条需求只改剧本，会动到几层？」** 理想答案是：**主要动配置与展示映射，少动壳与库。**

---

## 4. 与 AI 协作时的注意点（本层）

- 让 AI **先改模拟模块与对话面板**，不要轻易扩散到全局壳组件。
- 提示词中写明：**阻塞类型字符串** 与 **面板分支** 必须一致，否则会出现「永远不继续」或「渲染为空」。
- 新增一栏页面时，沿用 **页面模块** 模式，不要在 `App` 根组件堆业务逻辑。

---

## 5. 小结

| 问题 | 结论 |
|------|------|
| 模拟逻辑放哪 | 独立模块，与消息 UI 分离 |
| 剧本放哪 | 集中阶段配置，便于调延时与文案 |
| 库组件能改吗 | 默认不改；业务层扩展优先 |
