## 1. 背景与目标

### 1.1 背景

企业内需要一种典型的 **AI 工作台** 布局：用户从左侧管理或选择数据来源，在中间与智能助手对话，在右侧查看工作空间产出。本案例作为 **AI Coding 实验**，在既定组件库与设计约束下，验证复杂 UI、流式对话与多步骤交互的可实现性。

### 1.2 目标

1. 提供可运行的 **三栏工作台（UWS）** 骨架：顶栏 + 左「数据来源」、中「对话」、右「工作空间」。
2. 对话区支持 **模拟** 真实体验：流式输出、关键节点 **阻塞** 等待用户操作后再继续。
3. 展示 **多步骤思考/执行过程**（如分步加载与完成态），便于产品演示与后续对接真实接口。
4. 不追求生产级完备性，但需 **结构清晰**，便于将模拟层替换为真实 API。

### 1.3 非目标（本期不做）

- 真实用户体系、权限与多租户。
- 后端对话 API、RAG、文件实际上传与持久化。
- 无障碍与国际化专项达标（可在后续迭代补充）。
- 修改底层业务组件库 / shadcn 等 **通用层** 默认实现（除非经评审且属于通用能力）。

## 2. 用户与场景

### 2.1 目标用户

- 内部产品/设计：评审布局与交互原型。
- 研发：在相同技术栈上扩展真实业务。

### 2.2 核心场景

| 编号 | 场景 | 说明 |
|------|------|------|
| S1 | 首次进入 | 用户打开应用，看到顶栏标题与三栏布局；对话区展示欢迎/空状态与建议入口。 |
| S2 | 发起对话 | 用户输入或点击建议后，助手按预设剧本 **流式** 回复。 |
| S3 | 待办确认 | 流式介绍结束后出现 **待办清单**，用户确认后继续。 |
| S4 | 模板/流程确认 | 出现 **确认面板**（如模板选择），用户确认后继续。 |
| S5 | 补充信息 | 出现 **补充信息表单**，用户提交后继续。 |
| S6 | 长流程步骤 | 思考/执行步骤分步展示（加载态、完成态），节奏可读。 |
| S7 | 产出展示 | 工作区或对话内展示报告卡片等 **结果摘要**，并可有后续引导文案。 |
| S8 | 布局适配 | 左侧栏支持折叠/弹出等，保证主要操作路径可用。 |

## 3. 功能需求

### 3.1 布局与导航

- **F1** 顶栏：展示产品/助手名称、主要操作入口（如协作）、用户区占位。
- **F2** 三栏：左「数据来源」、中「对话」、右「工作空间」；各栏标题与主体区域区分清晰。
- **F3** 窄屏或折叠策略：以宿主实现为准，需保证对话区最小可用宽度。

### 3.2 对话能力（演示）

- **F4** 消息列表：用户消息与助手消息分区展示；助手消息支持 Markdown 等富文本能力（依赖现有组件）。
- **F5** 流式输出：助手文本按字符/块逐步出现，参数（速度、步长）可配置。
- **F6** 阻塞点：在指定剧情节点暂停，展示 `task_list` / `confirm_panel` / `supplement_form` 等交互，用户确认后 **同一剧本** 继续执行。
- **F7** 思考步骤：多步任务以可视化步骤组件展示，步骤间可有延时与状态变化。

### 3.3 数据来源与工作空间（演示）

- **F8** 左侧：展示数据来源相关 UI（列表、上传区等），可与真实数据解耦，以静态或 Mock 为主。
- **F9** 右侧：展示工作空间相关占位或报告卡片等，与对话剧情可联动。

---

## 4. 交互与体验要求

- **E1** 流式与阻塞切换自然，无重复触发或状态错乱。
- **E2** 阻塞点必须有明确主操作（确认/提交），避免用户无法继续。
- **E3** 关键文案为中文；演示剧本可与业务域文案一致。
- **E4** 视觉遵循设计系统：优先语义化 Token，避免硬编码色值（业务层可通过 `className` 微调）。

---

## 5. 技术约束（概述）

- 技术栈示例：React、TypeScript、Vite、Tailwind、业务 composed 组件 + shadcn/ui 等（以宿主工程为准）。
- 组件优先级示例：高层 composed → 低层 blocks → 基础 ui；尽量不修改通用组件库源码。
- 对话模拟逻辑与 UI **解耦**：建议独立模块（如「chat-simulation」），便于替换为真实接口。

| 编号 | 验收项 | 验证方式 |
