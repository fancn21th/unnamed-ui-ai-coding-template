# 需求实践清单

在进入设计与开发前，建议逐项确认。详细需求见同目录《需求文档-智能助手工作台》（若你重命名文件，请自行对应）。

**AI 辅助方式**：可用对话让 AI **根据清单逐项生成草稿**（如场景表、验收表），人类在编辑器中勾选与删改；最终签字仍以人类为准。

**工具对照（需求闸门）**：PRD 定稿前，建议在对话中要求 AI **遵守宿主项目已配置的 Rules**（技术栈、组件层级）；若将启动设计，在需求中写明是否使用 Figma 与 MCP（含义见本套顶层《README》中的工具索引表）。

---

## 1. 范围与优先级

- [ ] 已写明 **目标** 与 **非目标**，且干系人达成一致。
- [ ] 核心用户场景（如 S1–S8）已编号，且每条可映射到界面或流程。
- [ ] **演示 vs 生产** 边界已写清（接口、登录、数据持久化等）。

## 2. 可验收性

- [ ] 每条 P0 功能有 **可观察的验收方式**（手工步骤或自动化）。
- [ ] 阻塞类交互有明确 **完成条件**（按钮、表单校验等）。

## 3. AI 协作输入（含 Rules / Skills / MCP）

- [ ] 已准备或约定：**Figma** 文件与关键节点（若有视觉稿）；设计阶段可用 **Figma 类 MCP**（注意选区与 token 限制，细则以各团队内部文档为准）。
- [ ] **组件库 MCP** 已在宿主环境配置可用，或已约定改用内部组件清单文档手工查 API。
- [ ] 需求中的技术描述与 **Rules** 一致：至少核对「项目概述」「组件修改约束」是否与「少改底层组件」等非目标一致。
- [ ] 若用 AI 扩写需求，已提示其参考 **「生成页面」类 Skill** 的拆解顺序，避免跳到实现细节而缺场景。

## 4. 追溯

- [ ] 需求文档已定稿并存档、可检索。
- [ ] 重大变更经确认后更新文档，并可在评审记录中引用。

---

**完成标准**：P0 项全部勾选，且需求文档已评审存档。
